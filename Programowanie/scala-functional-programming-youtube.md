### Podstawy Programowania Funkcyjnego (5-7 minut)

**[Przejście do slajdu z tytułem: "Podstawy Programowania Funkcyjnego"]**

W tej części przyjrzymy się fundamentom programowania funkcyjnego. Co sprawia, że jest ono tak wyjątkowe i dlaczego warto się nim zainteresować?

**[Slajd z kluczowymi koncepcjami: Niemutacyjność, Funkcje Wyższego Rzędu, Rekurencja]**

Na początek, niemutacyjność. W programowaniu funkcyjnym, dane są niezmiennicze. Oznacza to, że zamiast modyfikować istniejące struktury danych, tworzymy nowe. 

**[Slajd z przykładem kodu Scali]**

```scala
val lista = List(1, 2, 3)
val nowaLista = lista.map(_ * 2)
```

Jak widać, zamiast zmieniać 'lista', tworzymy 'nowaLista'. To proste, ale potężne podejście pozwala uniknąć wielu błędów związanych ze stanem.

Następnie, funkcje wyższego rzędu. Są to funkcje, które mogą przyjmować inne funkcje jako argumenty lub zwracać je jako wynik.

**[Kolejny slajd z przykładem kodu]**

```scala
def aplikujFunkcje(f: Int => String, wartość: Int): String = f(wartość)
```

Tutaj 'aplikujFunkcje' przyjmuje funkcję 'f' i wartość, a następnie zwraca wynik zastosowania 'f' do tej wartości.

Na koniec, rekurencja, szczególnie rekurencja ogonowa, jest kluczowa w programowaniu funkcyjnym.

**[Slajd z przykładem rekurencji ogonowej w Scali]**

```scala
def sumaRekurencyjna(lista: List[Int], akumulator: Int = 0): Int = {
  lista match {
    case Nil => akumulator
    case x :: xs => sumaRekurencyjna(xs, akumulator + x)
  }
}
```

Ta funkcja sumuje elementy listy bez konieczności stosowania pętli czy mutowalnego stanu.

### Strict versus Non-Strict Evaluation (7-10 minut)

**[Przejście do slajdu z tytułem: "Strict versus Non-Strict Evaluation"]**

Kolejnym ważnym tematem w programowaniu funkcyjnym jest różnica między ewaluacją ściśle określoną, zwaną też 'strict', a leniwą, zwaną 'non-strict'. Rozumienie tej różnicy jest kluczowe, gdyż wpływa na sposób, w jaki nasz kod jest wykonany i zarządzany.

**[Slajd z definicją Strict Evaluation]**

W ewaluacji ściśle określonej, wyrażenia są obliczane natychmiast po ich zdefiniowaniu. To podejście jest bardziej intuicyjne, ale może prowadzić do niepotrzebnego obliczania wartości.

**[Slajd z przykładem Strict Evaluation w Scali]**

```scala
val wynik = List(1, 2, 3).map(x => x * 2)
```

Tutaj, wynik mapowania jest obliczany od razu, generując nową listę.

**[Przejście do slajdu z definicją Non-Strict Evaluation]**

Z drugiej strony, w ewaluacji leniwej, obliczenia są odroczone aż do momentu, gdy są faktycznie potrzebne. 

**[Slajd z przykładem Non-Strict Evaluation w Scali]**

```scala
lazy val wynik = List(1, 2, 3).map(x => x * 2)
```

W tym przypadku, 'wynik' nie zostanie obliczony, dopóki nie zostanie wykorzystany.

**[Slajd z porównaniem obu metod]**

To podejście jest szczególnie przydatne w dużych obliczeniach i strumieniowym przetwarzaniu danych, gdzie nie chcemy obliczać wszystkiego na raz.

**[Wróć do widoku gospodarza]**

Zrozumienie, kiedy stosować strict a kiedy non-strict evaluation, jest kluczowe dla efektywnego wykorzystania możliwości programowania funkcyjnego.
